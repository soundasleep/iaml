An IAML model instance may contain any number of {@model Session}s. The {@event onInit} {@model Event} for a {@model Session} is executed when a {@model Session} is accessed for the first time by a user, in the following way.

Every visitor to a web application is assigned a unique identifier in order to promote them to a user. If a {@model Session} is accessed by a user and no association exists between the users' unique identifier and the given {@model Session}, then a new relationship is created and stored as part of the {@model InternetApplication}. The current user now owns that {@model Session}. When this occurs, the \event{onInit} event for that {@model Session} is triggered.

If the current user restarts their web browser, the user will lose their unique identifier and will be demoted back into a visitor. Once a unique identifier has been lost, there is no way to regain access to a particular {@model Session}, and a new {@model Session} relationship will have to be defined. Sessions may also time out after a specified period of activity, and this period of activity should be configurable by the model developer.

The storage semantics for a {@model Session} are as follows: For a {@model Value} directly contained within a {@model Session}, any stored value is only accessible to the user that owns the containing {@model Session}. Once the {@model Session} ownership has been lost, any stored value is no longer accessible. Similarly, the values stored within a {@model DomainIterator} (including the current instance pointer) are only accessible to the current user.
